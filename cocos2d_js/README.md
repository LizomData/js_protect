# cocos2d js 保护方案V1

## 一、技术背景

- 出现游戏替换资源提取资源情况
- 游戏跳转链接被替换

## 二、方案需求

- 游戏资源保护：防提取防替换
- 跳转链接校验保护
- 防ai逆向：ai对抗
- 防人工逆向
- 消磨攻击者的耐心

## 三、保护方案概览

#### 3.1 静态防护

- 游戏资源加密保护
  - xxtea资源加密
  - 资源文件哈希打乱
  - 单文件单key解密，依赖运行时解密
  - 混入虚假资源数据
- 代码混淆
  - 解密壳代码采用高强度混淆
  - 交叉混淆游戏代码和解密壳代码

#### 3.2 动态防护

- 资源解密校验+防dump校验
  - 帧堆栈检测->防止直接调用解密函数dump
  - 资源key校验->非游戏正常调用解密就触发异常
  - 虚假资源陷阱->正常游戏不会解密虚假资源，只有攻击者才会选择dump所有资源
- 游戏资源完整性校验
  - 对所有存在的资源预设计算哈希值
  - 游戏运行时对预设资源进行哈希校验
- 防剥离
  - 游戏代码和壳代码交叉混淆
  - 游戏代码注入解密壳依赖
- 游戏埋雷
  - 概率跳转我们自己游戏链接
  - 触摸事件概率失灵
  - 游戏概率随机崩溃
- 浏览器环境校验
  - 防止node执行dump->ai习惯用node进行dump
  - 加反调试代码->防调试代码

## 四、保护方案细节描述

### 4.1 游戏资源加密保护

#### cocos2d资源格式解析细节

- 大部分资源在zip压缩
- zip压缩中存在__res的json资源文件，这个需要解析并转成文件 (正常游戏会转成全局明文资源对象)
- 暴露在html的全局资源对象，例如window.__res["icon"]="aaa"等

#### XXTEA 对称块加密

采用XXTEA加密算法，操作在 Uint32Array 上。**实现目的：**纯算术运算（移位、异或、加法），JS 里无需额外依赖；代码体积极小（加密 + 解密共 ~60 行），适合内嵌到 HTML；对短数据效率高（cocos 资源文件通常 1-100KB）；非标准算法（相比 AES），攻击者需要先识别算法种类

#### 压缩算法

采用Deflate对解压后的文件压缩。**实现目的：**控制体积膨胀。如果不先压缩，base64 后体积膨胀 2.5x+。先压缩再加密，体积接近原始大小（实测膨胀 < 3%）；提升安全性，压缩后的数据没有明显结构（JSON 的 `{` / `"` 等特征被消除），增加了已知明文攻击的难度

#### 每文件独立密钥（密钥派生）

每个文件单独计算加密密钥。**实现目的：**每个文件密钥不同，破解一个文件不等于破解全部；文件名参与密钥派生 → 攻击者不能把 A 文件的密文搬到 B 文件位置；攻击者即使逆向出 XXTEA + 密钥派生算法，还需要**`全局密钥种子`**才能解密任何文件

#### 文件名混淆（Hash）

目录层级完全铺平，攻击者无法从文件名推断内容类型，为后续埋雷做铺垫->防止运行时dump措施。

加密前 ZIP 文件列表:

```Plain
assets/main/native/4d/4ddf...png              ← 暴露资源类型
cocos-js/cc.js                                ← 暴露引擎本体
src/chunks/bundle.js                          ← 暴露业务代码位置
```

加密后 ZIP 文件列表:

```Plain
3a8f1b2c9d4e5f60      ← 无法判断文件类型
7e2d9a8c1b3f4e5d      ← 无法判断目录结构
9c1f3e8d7a2b4c5e      ← 无法判断是引擎还是资源
```

#### 虚假资源

俗称蜜罐 trap 文件（随机 hash 名 + 随机密文，混入 ZIP 中与真实文件无法区分）。**实现目的：**真实文件都通过哈希混淆，攻击者无法判断哪个是虚假的资源文件，一旦dump虚假的资源，就会触发非法dump操作->防ai一键dump会很有效果，ai一般会采用无头浏览器一键dump，dump时就会触发蜜罐陷阱，资源将膨胀到无头浏览器卡死，甚至ai卡死

#### 虚假资源注入，dump就检测到非法

游戏本身逻辑不会对虚假资源的key进行调用，攻击者拿到所有资源key，想通过界面函数导出全部资源就会触发假资源检测

#### 帧堆栈检测->防常规调用dump

攻击者采用常规导出调用解密函数(跟游戏本身调用解密逻辑存在差异)就会触发非法dump的检测

#### 单文件单密钥，防止密钥碰撞解密

防止密钥碰撞，加大解密难度

### 4.3 游戏完整性校验方案

#### 对游戏资源进行哈希校验

对res资源对象进行哈希计算，预先存入游戏本身的哈希值，运行游戏时进行动态计算校验，计算出来的哈希值不对就触发游戏完整性保护

#### 跳转链接校验

挂钩浏览器所有跳转逻辑，对跳转链接进行校验，不是我们游戏的跳转链接就触发非法操作

### 4.4 游戏埋雷方案

#### 概率跳转我们自己游戏链接

对应非法改跳转链接的操作，设置概率跳转游戏自身链接

#### 触摸事件概率失灵

完整性校验失败后开启游戏触摸失灵

#### 游戏概率随机崩溃

完整性校验失败后游戏随机崩溃

### 4.5 防剥离方案

#### 解密壳和游戏逻辑交叉混淆

解密壳和游戏逻辑代码交叉融合，防止剥离正常的游戏逻辑或者解密壳：

解密器高强度混淆->合并游戏代码->通过low混淆交叉融合

#### 在原游戏代码注入壳依赖环境 (就算dump出来js代码也执行不了)

对于剥离壳的操作，因为缺乏壳的环境初始化，js代码可以预设执行失败：采用散布验证暗桩

解密器 IIFE 里:

```JavaScript
// 设置一系列隐蔽的验证点（随机变量名、随机值）
var _a7f = 0xB3C2;
var k9d = 'x' + String.fromCharCode(72, 101);
window['$' + _a7f.toString(36)] = _k9d;
```

加密时在游戏代码的随机位置注入:

```JavaScript
// 暗桩 1（看起来像正常赋值后的空检查）
if (!window['_$' + (0xB3C2).toString(36)]) { setTimeout(function(){ document.body.innerHTML=''; }, 5000); }

// 暗桩 2（藏在 try-catch 里）
try { var _t = window['_$' + (0xB3C2).toString(36)]; _t.charAt(0); } catch(e) { location.href = 'about:blank'; }

// 暗桩 3（延迟执行，不立即崩溃）
setTimeout(function() {
      if (typeof window['_$' + (0xB3C2).toString(36)] === 'undefined') {
        // 5秒后静默破坏画布
        var c = document.querySelector('canvas');
        if (c) c.getContext('2d').clearRect(0,0,9999,9999);
      }
    }, 8000);
    
```

  优点：

- 多个暗桩散布在不同位置，攻击者要全部找到并移除
- 每个暗桩的检查方式不同（直接判断/try-catch/setTimeout）
- 触发方式不同（立即崩溃/延迟清空/跳转）
- 变量名和值每次加密都随机，无法硬编码 bypass
- 混淆后暗桩和正常游戏代码外观无法区分

  缺点：需要在游戏代码中找安全的注入位置（不能插在表达式中间）

## 五、测试方案

### 5.1 性能测试

对混淆后的游戏和混淆前的游戏进行性能测试，主要测试方向为：

- 游戏打开时间
- 游戏游玩是否卡顿
- 链接跳转是否流畅正常

### 5.2 防护程度测试

采用claude code opus4.6进行逆向分析，防护强度直接跟ai逆向分析所花token和时间成正比，需要关注ai逆向的策略：

- 静态分析
  - 对代码进行反混淆
  - 定位关键字符串
  - 定位关键逻辑
- 动态分析：
  - 采用node进行dump
  - 采用无头浏览器进行dump
